# Review Guidance

- Reviews are adversarial research critiques of ideas, questions, experiment plans, claims, decisions, or paper drafts.
- Review notes should identify the strongest strengths and weaknesses, especially unclear contribution, weak novelty, missing baseline, invalid metric, confound, unsupported claim, scope mismatch, or likely reviewer rejection.
- Reviews are not approval gates or planning sessions; they pressure-test the work so its risks are explicit.
- Each review should end with a concise verdict: main strengths, main weaknesses, most serious objection, and suggested follow-up if any.
- Link reviewed questions, experiments, claims, decisions, literature, configs, results, and paper sections when useful.
- Name review files `YYYY-MM-DD-short-title.md` using the date the review is recorded.
- Keep category-level summaries in `README.md` factual; leave it title-only until there is something real to summarize.
- Track review files from the category dashboard as `[review](YYYY-MM-DD-review-title.md): Open`, then replace `Open` with the verdict or linked follow-up.
